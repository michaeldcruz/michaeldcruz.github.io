> Draft. The spec-as-source-of-truth post. Practitioner-facing.

## The thesis

When agents build the routine change, the thing you actually author is the spec. It goes in, working software comes out. That reframes documentation from an afterthought into the primary artifact. A stale spec isn't a paperwork problem anymore, it's a corrupted input. The discipline of keeping the spec true to the running system is the new core engineering skill.

## Beats to develop

- The shift: humans stop writing the implementation and start writing the intent. The intent is now load-bearing.
- Why a stale spec is worse in a factory than in a human shop. A human engineer silently corrects for outdated docs. An agent builds exactly what the lie says.
- What "living" requires. The spec updates as the system changes, and there's a mechanism that catches drift between spec and reality.
- The payoff. A spec that's actually trustworthy is the highest-leverage thing you own, because it's the thing that becomes software.

## The line

A living spec goes in, tested working software comes out. The word doing the work in that sentence is "living."

## So what

Look at your specs and ask: if an agent built exactly what this says, with no judgment, would you get the system you have? The gap between "yes" and "no" is your real backlog.
